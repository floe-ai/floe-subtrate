# Architecture Integration Brief: issue-51-inline-scope-creation

## Existing ownership

- Package/component/module/library: FloeWeb owns the Workspace Home assignment affordance in `floe-web\src\main.tsx`, pure Workspace Context eligibility/copy in `floe-web\src\contexts.ts`, Scope/Field client helpers in `floe-web\src\scope-projection-api.ts`, and audited assignment client helpers in `floe-web\src\context-assignment-api.ts`. Bus owns canonical Scope creation and Context-to-Scope membership in `floe-bus\src\server.ts`, `floe-bus\src\store.ts`, and `floe-bus\src\scopes\store.ts`.
- Current owner rationale: Workspace Home already renders real Scopes as Fields, lists unscoped actor-anchored Contexts separately, lets a user assign one of those Contexts to an existing named Scope, and refreshes the target Scope Projection. Bus already exposes `POST /v1/workspaces/:workspace_id/scopes` for real named Scope creation and `POST /v1/workspaces/:workspace_id/contexts/:context_id/assign-scope` for audited assignment; FloeWeb must compose those two existing APIs rather than inventing a parallel Field-owned membership path.
- Source evidence: `floe-web\src\main.tsx` imports `createScope`, `listScopes`, `assignContextToScope`, `canAssignContextToScope`, and dialog helpers; tracks `scopeRecords`, `fieldSummaries`, `workspaceContexts`, and `contextAssignmentTargets`; `assignWorkspaceContextToScope` calls `assignContextToScope` then refreshes Contexts and the target Field. `floe-web\src\scope-projection-api.ts` exposes `createScope`, `listScopes`, `getScopeProjection`, and layout sidecar helpers against `/scopes`, not `/fields`. `floe-bus\src\server.ts` handles Scope creation at lines 178-213 and assignment at lines 732-781. `floe-bus\src\store.ts` broadcasts `scope_created` from `createScope` and writes `context.scope_assigned` from `assignContextScope`. `floe-bus\src\scopes\store.ts` reserves id `default`, stores `is_default = 0`, and returns real Scope records. `CONTEXT.md` and `docs\adr\0004-scope-as-substrate-organising-boundary.md` define Field as a Scope projection and reject Default Scope/Default Field behavior.

## Existing interaction model

- User/system behaviors that already exist: Workspace Home shows a Fields pane derived from real Scope records and a separate `Workspace-level Contexts` pane for unscoped actor Context streams. A user can open a Workspace-level Context without scoping it, choose an existing named Scope from a row-level select, and assign via the audited Bus assignment API. Successful assignment removes the Context from the Workspace-level list and the target Field projection renders the same Context through Scope Projection. Field creation from Workspace Home currently uses the `New Field` prompt, creates a Scope through `createScope`, refreshes Fields, opens the new Field, and loads its projection.
- Behaviors that must remain unchanged: Workspace Home must remain an index/dashboard, not a Scope. Opening direct Contexts must continue to use the actor conversation sidebar. Existing assignment to an existing named Scope must continue to require an explicit Scope selection and must not assign scoped or actorless orphan Contexts. Field/canvas behavior must continue to come from Scope Projection plus renderer layout sidecars; assignment must not directly insert React Flow nodes or Field membership. Field creation outside assignment must still create a real Scope-backed Field.
- Runtime or UX evidence: `renderHome` renders Fields and Workspace-level Contexts as separate sections; `workspaceContextLabel` and `contextScopeAssignmentStatus` avoid Default wording; `canAssignContextToScope` returns true only for unscoped Contexts with participants. `projectionToReactFlow` builds Context nodes from `projection.refs.contexts`, so a newly assigned Context appears only after the Bus projection includes it. The dialog system supports text input validation and async `onConfirm` failures with inline error text; `no-native-popups.test.ts` prevents native alert/confirm/prompt usage.

## Existing extension points

- APIs/hooks/components/library features/stores/conventions to use: Use `createScope(busUrl, workspaceId, { title })` for Scope creation and `assignContextToScope(busUrl, workspaceId, contextId, { scopeId, actorId })` for assignment. Use `promptDialog`/`DialogHost` input validation and async `onConfirm` to collect and validate the new Scope name and surface creation/assignment failures inline. Use existing `refreshFields`, `refreshContexts`, `refreshWorkspaceContexts`, and `refreshOpenField` after success; update `contextAssignmentTargets` only as local UI state, not semantic state. Use existing `ScopeProjectionApiError` and `ContextAssignmentApiError` messages where possible.
- Relevant docs or library capabilities: React Flow already owns canvas interactions through `<ReactFlow>`, `onNodesChange`, `onConnect`, `onDrop`, `Controls`, `MiniMap`, handles, and node actions. This slice should be a Workspace Home/dialog composition; canvas rendering must be produced by the existing Scope Projection refresh.
- Existing examples in this codebase: `promptCreateField` shows `promptDialog` + `createScope` + `refreshFields` + `refreshOpenField`; `assignWorkspaceContextToScope` shows audited assignment + Context/Field refresh; `scope-projection-api.test.ts` covers encoded Scope creation and error surfacing; `context-assignment-api.test.ts` covers assignment URL/body and mapped errors; `floe-web\tests\helpers.ts` already mocks Scope POST, assign-scope POST, and projection mutation for Playwright vertical slices.

## Do-not-bypass list

- Systems/libraries/components not to duplicate or replace: Do not create a Field-owned item list, Field-owned connection graph, `.floe/blocks` substrate, direct React Flow node insertion, direct SQLite/client state mutation, replacement Context, copied Events, or legacy `/fields` endpoint call. Do not bypass Bus `createScope` or Bus `assignContextScope`. Do not use native browser popups.
- Shortcuts or parallel paths to avoid: Do not silently create a Scope when normal assignment fails. Do not auto-create a hidden fallback Scope. Do not route through `default`, `Default Scope`, or `Default Field`. Do not infer Scope membership from a Field title. Do not implement assignment by changing only the currently loaded projection/layout. Do not add reassignment for already-scoped Contexts unless a separate slice defines it.
- Invariants: Workspace Home is not a Scope; actors are Workspace-level identities; Contexts are streams; actor-anchored direct Contexts may be unscoped; actorless/event-linked operational Contexts require Scope; no Context may exist with neither actor participants nor Scope; Field/FloeWeb is not the source of Scope membership; Scope membership comes from substrate primitives; no fake Default Scope hides modeling ambiguity. FloeWeb Field/canvas invariants: use existing React Flow features before hand-rolling interactions; preserve Block Library drag/drop; preserve node icons, labels, handles, selection, pan, zoom, drag, rename/open, and connection affordances; substrate-backed behavior must integrate into the existing Field/canvas model; toolbar or dialog shortcuts may supplement canvas flows but not replace them; pan/zoom/drag/navigation performance regressions are blockers.

## Integration plan

- Insert the change at: Add an inline "create Scope and assign" affordance in `renderHome` inside each eligible `.workspace-context-block`, next to the existing Scope select / Assign control. Add a handler near `assignWorkspaceContextToScope` that validates `canAssignContextToScope(context)`, opens `promptDialog` for a new named Scope, calls `createScope`, then calls `assignContextToScope` with the returned `scope.scope_id`, then refreshes Workspace Home, Field list/Scope records, participant Contexts, and target/open Field projection.
- Why this is the correct integration point: Workspace Home already has the selected Workspace, self actor id, eligible Context row, Scope records, dialog helpers, and refresh functions in scope. Bus already owns both state transitions. Composing those APIs at the row-level affordance keeps creation explicit, keeps assignment audited, and lets Scope Projection render the assigned Context as a Field node after refresh.
- Alternatives considered and rejected: Rejected modifying `createField` directly as the only path because it currently navigates immediately and returns `void`; inline assignment needs the created `ScopeRecord` before assignment. If reused, factor a small internal `createScopeRecord` helper that returns the Scope and let `createField` keep its existing behavior. Rejected using a free-text Scope id for assignment because Scope identity should come from the created Scope response. Rejected rollback by deleting the new Scope if assignment fails because Scope deletion is not available and deleting a real Scope would be a new behavior. Rejected creating a temporary projection node before refresh because Field membership must come from Bus Scope Projection.

## Regression checklist

- Behavior: Existing assignment to an existing named Scope still posts only to `/assign-scope`, preserves `context_id`, removes the row from Workspace-level Contexts, and renders the Context through the chosen Scope/Field projection.
- Behavior: Inline creation posts to `/v1/workspaces/:workspace_id/scopes`, creates a real non-default Scope/Field, then posts to `/assign-scope` with the returned `scope_id`; no assignment occurs if creation fails.
- Behavior: Blank/whitespace names and misleading Default/Default Scope/Default Field names are blocked in UI validation before Scope creation. Bus still rejects reserved id `default`; UI must not rely on a generated or hidden Default.
- Behavior: Scope creation or assignment failure is visible and recoverable in structured UI/dialog error state; if Scope creation succeeded but assignment failed, the real new Scope remains visible after refresh and the failure is not hidden.
- Behavior: Workspace Home, Field list/Scope records, participant Context list, target Field projection, and any currently open target Field refresh consistently after success.
- Behavior: Opening Workspace-level Contexts and projected Context nodes continues to use the existing conversation sidebar.
- Behavior: React Flow Field/canvas pan, zoom, drag, selection, handles, node labels/icons, node Open, rename, Block Library drag/drop, pulse-context connection creation/deletion, and layout persistence remain unchanged.

## Risk assessment

- Risk: The current Bus Scope creation API accepts any non-empty title and does not trim or reject misleading names; UI validation must enforce acceptance criteria, while server-side hardening can be a separate follow-up unless this slice chooses to add it deliberately.
- Risk: The existing `createField` helper defaults blank names to `Field N`; reusing it directly would violate "prevents blank" for inline Scope creation and would navigate before assignment. Prefer a new chain handler or factored helper that returns `ScopeRecord`.
- Risk: If Scope creation succeeds and assignment fails, users may see a new empty Field and an unassigned Context. This is honest substrate state; surface the failure clearly, refresh Fields, and let the user retry assignment to the new real Scope rather than hiding or rolling back.
- Risk: Adding controls inside Workspace Context rows can create nested click/focus/accessibility issues or accidentally open the conversation while creating/assigning. Use sibling controls, explicit button labels, and Playwright assertions.
- Risk: Refresh ordering can produce stale UI if `refreshFields` is skipped or if `openField` races `refreshOpenField`. Use the created `scope_id`, await refreshes, and update/open the target Field only after assignment success.
- Risk: Existing test helper `seedApp` still creates a mocked `default` Scope named `Default` for older specs, while current docs say Default Scope is not product behavior. Treat this as stale fixture compatibility; do not copy it into new tests, and flag visible Default UI as a regression for this slice.
- Mitigation: Keep the implementation narrow in Workspace Home and existing API/dialog/refresh paths; add tests for request ordering, no assignment after creation failure, validation copy, no Default terminology, and projection-based rendering.

## Decision confidence

- Confidence: high
- Reasons: Both required substrate APIs already exist and are covered; FloeWeb already has typed helpers, row-level assignment UI, Scope creation UI, dialog validation/error mechanics, and Playwright route support for Scope POST plus assign-scope POST. The architecture choice is composition of existing owners, not new substrate design.
- Open questions: Product copy/placement can vary: recommended default is a row-level "Create Scope and assign" action that opens a `Create Scope for Context` dialog and navigates/opens the new Field after successful assignment, matching current assignment behavior. If the product prefers staying on Workspace Home with a success message and an "Open Field" affordance, that is also compatible, but implementation should choose one behavior and test it.
