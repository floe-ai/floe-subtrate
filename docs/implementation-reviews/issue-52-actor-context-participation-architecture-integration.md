# Architecture Integration Brief: issue-52-actor-context-participation

## Existing ownership

- Package/component/module/library: FloeWeb owns the actor conversation/sidebar and Workspace Home presentation in `floe-web\src\main.tsx`; pure Context sorting/label/status semantics live in `floe-web\src\contexts.ts`; FloeBus owns canonical Context discovery, Context events, participant membership, Scope anchoring, and assignment in `floe-bus\src\server.ts`, `floe-bus\src\contexts\store.ts`, `floe-bus\src\contexts\resolver.ts`, and `floe-bus\src\store.ts`.
- Current owner rationale: The actor sidebar already selects workspace-scoped Actors and opens Context streams by setting `selectedContextId` and loading `/v1/contexts/:id/events` (`main.tsx` lines 718-739, 1008-1014, 2251-2525). Bus already exposes participant-based discovery through `GET /v1/contexts?participant=...&workspace_id=...` and returns both scoped and unscoped Contexts unless `scope_id` is supplied (`server.ts` lines 687-702; `contexts\store.ts` lines 220-235). Workspace Home already owns unscoped Workspace-level discovery and #50 assignment affordances through `GET /v1/workspaces/:workspace_id/contexts?scope=unscoped` and `assign-scope` (`main.tsx` lines 686-696, 741-766, 1884-1955).
- Source evidence: `CONTEXT.md` says Workspace is top-level, Workspace Home is not a Scope, Actors are workspace-scoped, Contexts are streams, actor-participant Contexts may be unscoped, and Default Scope/Default Field terminology is forbidden (lines 9-15, 71-77, 131-149, 159-164). `docs\adr\0004-scope-as-substrate-organising-boundary.md` corrects the Default Scope assumption and states unscoped actor Contexts remain discoverable through Workspace, Actor, and Context views rather than a fake Default Field (lines 15-53). The #50 brief identifies the same owners for Workspace Home assignment and warns not to bypass Bus assignment, Scope Projection, dialogs, or React Flow.

## Existing interaction model

- User/system behaviors that already exist: The actor conversation panel opens from the global sidebar button, shows a workspace-scoped Actor selector, lists Contexts under `Conversations`, selects a Context by `context_id`, fetches events through the existing Context events API, and renders messages/activity/pulse cards in one stream view (`main.tsx` lines 2251-2525). Workspace Home separately lists unscoped `Workspace-level Contexts` and opens them with `openWorkspaceContext`, including fallback actor selection when the current Actor is not a participant (`main.tsx` lines 725-739, 1884-1955). Field projected Context nodes open the same conversation/sidebar path through `openProjectedContext` (`main.tsx` lines 718-723, 768-800).
- Behaviors that must remain unchanged: Actors stay workspace-scoped and must not be rendered as Field/canvas membership or placed inside Scopes. Opening a listed Context must reuse `selectedContextId` + `/v1/contexts/:id/events` rather than a new stream reader. Workspace Home must remain an index/dashboard, not a Scope. Field/Scope Projection remains scoped-only and substrate-derived; it must not become the source of actor participation. Existing no-bleed protections must still prevent messages, telemetry, and stale event responses from one Context appearing in another.
- Runtime or UX evidence: `sortContextsForAgent` currently filters to Contexts where both operator and selected actor participate (`contexts.ts` lines 66-85), and `refreshContexts` currently queries only the operator participant (`main.tsx` lines 673-684). This is the main code/goal conflict: #52 asks selected Actors to surface Contexts they participate in across the Workspace, but current code/tests intentionally show only operator-involved Contexts. `no-actor-bleed.spec.ts` currently asserts the query uses the operator self id, not the selected Actor (lines 187-223); that test will need to be revised to preserve no-bleed semantics without preserving the old discovery query.

## Existing extension points

- APIs/hooks/components/library features/stores/conventions to use: Use `GET /v1/contexts?participant=${selectedActor.endpoint_id}&workspace_id=${workspaceId}` for selected-actor participation discovery because the Bus endpoint already supports participant-scoped Workspace Context lookup and includes both scoped and unscoped Contexts (`server.ts` lines 687-702). Keep `GET /v1/contexts/:id/events` for opening streams. Use existing `scopeRecords` from `listScopes` to resolve a scoped Context's Scope title for labels; use `scope_id === null` for `Workspace-level Context` labels. Extend pure helpers in `floe-web\src\contexts.ts` for sorting/filtering/labels rather than embedding model terminology directly in JSX.
- Relevant docs or library capabilities: React state/hooks already own sidebar selection (`selectedAgentId`, `contexts`, `selectedContextId`, `contextEventsState`). The existing Bus resolver rejects explicit emits into a Context where the source endpoint is not a participant (`contexts\resolver.ts` lines 89-96; `contexts\integration.test.ts` lines 190-207), so any actor-participation list that includes Contexts without the operator must disable or guard the composer for those Contexts instead of trying to send and relying on server errors.
- Existing examples in this codebase: `openWorkspaceContext` shows how to select an Actor participant before opening a Context (`main.tsx` lines 725-739). Workspace Home labels unscoped Contexts as `Workspace-level Contexts` and `Not assigned to a Scope` without Default terminology (`main.tsx` lines 1884-1955; `contexts.ts` lines 96-109; `contexts.test.ts` lines 177-210). `workspace-home-assignment.spec.ts` demonstrates seeded mixed Scope/Context routes and no Default Scope fallback (lines 11-55); `tests\helpers.ts` already has mutable Context fixtures and an assign-scope route that can be extended for #52 mixed scoped/unscoped sidebar tests (lines 384-445).

## Do-not-bypass list

- Systems/libraries/components not to duplicate or replace: Do not add a new Context stream renderer, alternate event fetch path, Field-owned membership list, client-only actor membership model, or local fake Context index. Do not use Scope Projection or React Flow nodes as the source of actor participation. Do not bypass Bus `context_participants`, `contexts.scope_id`, `/v1/contexts`, or `/v1/contexts/:id/events`.
- Shortcuts or parallel paths to avoid: Do not keep the old operator-only participant query if the UI copy says it is showing all Contexts for the selected Actor. Do not invent “Default Scope,” “Default Field,” “Home Scope,” “actor Scope,” or any copy implying an Actor lives inside a Scope/Field. Do not assign or move Contexts as part of viewing them. Do not auto-add the operator to a third-party selected-actor Context just so the composer works. Do not route scoped Context discovery through Field canvas projection.
- Invariants: Workspace is top-level; Actors are workspace-scoped identities and are not inside Scopes or Fields; Contexts are streams; actor-participant Contexts may have `scope_id: null`; actorless Contexts and Pulse/Webhook/event-source operational Contexts require a real non-null Scope; no Context may exist with neither actors nor Scope; Workspace Home is not a Scope; Field/FloeWeb must not determine Scope membership; no fake Default Scope should hide modeling ambiguity.

## Integration plan

- Insert the change at: Extend the existing actor sidebar flow in `renderChannel` and the pure helpers in `floe-web\src\contexts.ts`. Change `refreshContexts`/its caller so the sidebar list is driven by the selected Actor participant endpoint, not the operator endpoint, while preserving Workspace filtering. Add a helper that sorts selected-actor Contexts by activity and classifies each row as `Workspace-level Context` (`scope_id === null`) or `Scoped Context` with an optional real Scope title from `scopeRecords`. Update the list header/copy from generic `Conversations` to model-correct actor participation wording such as `Contexts involving {actorName}`.
- Why this is the correct integration point: The actor sidebar is already the Actor selection and Context stream owner. The Bus participant endpoint already returns exactly the substrate-owned participation rows needed across the Workspace; adding another API/store would duplicate ownership. `contexts.ts` is already where actor-neutral Context list semantics are unit-tested, so it is the right place to encode no-Default labels and mixed scoped/unscoped sorting.
- Alternatives considered and rejected: Rejected using Workspace Home's unscoped list because it intentionally excludes scoped Contexts and is not actor-specific. Rejected using Scope Projection because it is scoped-only and Field-facing, and would miss unscoped Contexts while implying actor membership comes from Fields. Rejected keeping the operator-only query with stronger copy because it would not satisfy the selected-actor participation goal. Rejected auto-opening Fields for scoped actor Contexts because acceptance requires the existing Context stream/conversation path, not Field navigation.

## Regression checklist

- Behavior: Selecting an Actor in the sidebar shows Contexts that selected Actor participates in across the current Workspace, including unscoped Workspace-level Contexts and scoped Contexts returned by Bus.
- Behavior: Mixed scoped/unscoped rows are labelled with `Workspace-level Context` and `Scoped Context`/real Scope names, with no `Default Scope`, `Default Field`, or actor-in-Scope wording.
- Behavior: Clicking any listed Context uses the existing `selectedContextId` and `/v1/contexts/:id/events` stream path and does not create, assign, copy, or re-anchor the Context.
- Behavior: If the opened selected-actor Context does not include the operator, the UI must not allow sending a message that Bus will reject as `E_NOT_CONTEXT_PARTICIPANT`; show a read-only/inspection state or otherwise clearly guard the composer.
- Behavior: Existing no-bleed protections remain: selecting one Context cannot show events, pulse cards, telemetry, streaming output, or activity from another Context.
- Behavior: Workspace Home #50 assignment, Field projected Context opening, Scope Projection, Field/canvas interactions, and actor-neutral wording remain intact.

## Risk assessment

- Risk: Switching discovery from operator participant to selected Actor participant intentionally conflicts with the current `no-actor-bleed` query assertion and can expose Contexts where the operator is not a participant. Mitigation: update tests to assert the new ownership and add a clear read-only/composer guard for non-operator Contexts.
- Risk: Existing `selectedContext` lookup only searches `contexts` and `workspaceContexts`; changing `contexts` semantics can affect Field-projected Context opening if the currently selected Actor is not a participant. Mitigation: keep `openProjectedContext` regression-tested and either preserve a fallback fetch/selection path for projected Contexts or ensure opening projected Contexts selects a participating Actor before setting the Context.
- Risk: Scoped labels may tempt implementers to navigate to Fields or render Scope membership. Mitigation: labels are metadata only; opening remains the Context stream path and Field/canvas remains untouched.
- Risk: Scope ids without loaded `scopeRecords` could degrade to raw ids or ambiguous copy. Mitigation: use existing `listScopes`/`scopeRecords`, fall back to `Scoped Context` without exposing fake defaults, and refresh scopes on workspace changes as today.
- Mitigation: Keep the implementation narrow in `contexts.ts` and `main.tsx`; use existing Bus APIs; add unit and Playwright tests for mixed lists, labels, stream opening, no Default wording, read-only guard, and no-bleed behavior.

## Decision confidence

- Confidence: medium
- Reasons: Ownership and existing extension points are clear: FloeWeb actor sidebar plus Bus participant/events APIs. The main ambiguity is product semantics: the acceptance wording implies selected-actor participation even when the operator is not a participant, while existing code/tests intentionally query by operator and filter to operator+selected actor. Implementation is straightforward if this behavior change is accepted, but it must deliberately revise the old no-bleed query invariant and guard composer behavior.
- Open questions: Should the #52 list include Contexts where the selected Actor participates but the operator does not? Recommended answer: yes, because that is the literal acceptance criterion, but open those Contexts as read-only unless/until the operator is a participant. If the intended scope is only operator↔selected-actor Contexts, revise the acceptance wording before implementation and keep the operator participant query.
