# Architecture Integration Brief: issue-50-workspace-home-assignment-affordance

## Existing ownership
- Package/component/module/library: FloeWeb owns the Workspace Home and actor conversation affordance in `floe-web\src\main.tsx`; pure Context list/label semantics live in `floe-web\src\contexts.ts`; Scope/Field projection client helpers live in `floe-web\src\scope-projection-api.ts` and projection-to-React-Flow mapping in `floe-web\src\scope-projection.ts`; Bus owns canonical Context Scope assignment in `floe-bus\src\server.ts`, `floe-bus\src\store.ts`, and `floe-bus\src\contexts\store.ts`.
- Current owner rationale: Workspace Home already lists unscoped actor Contexts by calling `GET /v1/workspaces/:workspace_id/contexts?scope=unscoped&limit=6`, opens them through the existing actor conversation sidebar, and renders Fields from real Scopes via Scope Projection. Bus already exposes the audited assignment endpoint `POST /v1/workspaces/:workspace_id/contexts/:context_id/assign-scope`; the client must call it rather than creating/copying Contexts.
- Source evidence: `CONTEXT.md` defines Workspace Home as an index/dashboard, Field as a Scope projection, Workspace-level Context as actor participants with no Scope, and invalid Contexts as having neither actor participants nor Scope. `docs\adr\0004-scope-as-substrate-organising-boundary.md` supersedes Default Scope behavior. `floe-web\src\main.tsx` imports `DialogHost`, `confirmWithOptions`, `prompt`, `listScopes`, `getScopeProjection`, and renders Workspace-level Contexts around lines 1788-1890. `floe-bus\src\server.ts` handles assignment around lines 732-781; `floe-bus\src\store.ts` preserves Context identity, updates `scope_id`, inserts `context.scope_assigned`, and broadcasts around lines 950-1008.

## Existing interaction model
- User/system behaviors that already exist: Workspace Home shows Fields from real Scope records and a separate `Workspace-level Contexts` section for unscoped actor Contexts; clicking a Workspace-level Context opens the existing actor conversation panel via `openWorkspaceContext`; double-clicking or clicking Open on a projected Context node opens the same conversation path via `openProjectedContext`; Fields open through `openField` and refresh with `refreshOpenField`; global confirmations use the app dialog system, not native popups.
- Behaviors that must remain unchanged: Workspace Home must not become a Scope or fake Field; actors remain workspace-scoped; unscoped actor Contexts remain valid until explicitly assigned; already-scoped Contexts remain visible only through their target Scope/Field projection and actor conversation list, not as assignable Workspace-level items; Scope Projection remains scoped-only; React Flow pan/zoom/drag/selection, node handles, node labels/icons, connection edit/delete, Block Library drag/drop, and Field rename/open affordances must continue working.
- Runtime or UX evidence: `floe-web\src\main.tsx` keeps `contexts` and `workspaceContexts` as separate state, derives `selectedContext` from both lists, and refreshes both on workspace/event changes. `floe-bus\src\scopes\projection.ts` builds projections from `listContextsForScope(workspaceId, scopeId)`, so unscoped Contexts are intentionally excluded. `floe-web\src\no-native-popups.test.ts` forbids `window.confirm`, `window.prompt`, and `window.alert` in app source.

## Existing extension points
- APIs/hooks/components/library features/stores/conventions to use: Use `api(...)` or an equivalent typed FloeWeb API helper to call `POST /v1/workspaces/:workspace_id/contexts/:context_id/assign-scope` with `{ scope_id, assigned_by, reason? }`; use existing `scopeRecords`/`fieldSummaries` populated by `listScopes`; use `confirmDialog`/`confirmWithOptions` or a small app-owned dialog extension for confirmation and inline failure; use `refreshWorkspaceContexts`, `refreshContexts`, `refreshFields`, and `refreshOpenField` for post-success refresh; use `openField(scopeId)` or set the Field view only after successful assignment if the chosen behavior is to navigate to the target projection.
- Relevant docs or library capabilities: React Flow already owns graph interaction through `<ReactFlow>` props (`onNodesChange`, `onConnect`, `onDrop`, `Controls`, `MiniMap`, handles). The assignment slice should not hand-roll canvas interactions; a Workspace Home/button/dialog affordance is enough, and any projection appearance must come from Bus Scope Projection refresh.
- Existing examples in this codebase: `deleteWorkspace` and `deleteConversation` show dialog `onConfirm` API calls with inline errors; `promptCreateField` shows app dialog input validation flow; `createField`, `renameField`, and pulse subscribe/unsubscribe flows show the pattern of optimistic/local updates followed by canonical Scope Projection refresh.

## Do-not-bypass list
- Systems/libraries/components not to duplicate or replace: Do not create a replacement Context, copy events, edit Field layout to imply membership, write to legacy Field endpoints, write directly to React Flow nodes as semantic state, or update Bus SQLite/client state directly. Do not add a hidden Default Scope/Default Field or route assignment through scope id `default`.
- Shortcuts or parallel paths to avoid: Do not implement reassignment for already-scoped Contexts; do not show assignment on projected/scoped Context rows as if reassignment exists; do not infer Scope from Field title text; do not create a second Workspace Home context store; do not use native browser popups; do not make assignment a message/event emit shortcut.
- Invariants: Context identity must be preserved; assignment is explicit and audited by Bus; `scope_id` changes only through the assignment API; Scope selection must choose an existing real Scope; Workspace-level Context list contains only `scope_id: null` actor Contexts; Scope Projection contains only real scoped substrate records. FloeWeb Field/canvas invariants: use existing React Flow features before hand-rolling interactions; preserve Block Library drag/drop unless explicitly redesigned; preserve node icons, labels, handles, selection, pan, zoom, drag, rename, and connection affordances; substrate-backed behavior must integrate into the existing Field/canvas model, not create a separate path; toolbar shortcuts may supplement canvas flows, not silently replace them; performance regressions in navigation, pan, zoom, or drag are blockers; existing working affordances require regression tests before refactor.

## Integration plan
- Insert the change at: Add the Workspace Home assignment affordance inside the `recentWorkspaceContexts.map(...)` rendering in `renderHome`, beside or within each eligible Workspace-level Context row. Add a handler near `openWorkspaceContext`/conversation actions that (1) verifies `selectedWorkspace` and `context.scope_id === null`, (2) requires a selected existing `ScopeRecord`, (3) confirms the operation with clear copy naming the Context and target Scope, (4) calls the existing Bus assignment endpoint, and (5) refreshes `workspaceContexts`, participant `contexts`, `fieldSummaries`/`scopeRecords`, and the target/open Field projection.
- Why this is the correct integration point: Workspace Home already owns discovery of unscoped Contexts and has the required `scopeRecords`, Workspace id, operator actor id, and refresh functions in scope. The Bus endpoint already owns validation, audit event creation, identity preservation, and projection eligibility. Refreshing Scope Projection after success lets the existing Field/canvas model render the assigned Context instead of inventing a parallel client projection.
- Alternatives considered and rejected: Rejected creating a new Context in the selected Scope because it violates identity/audit criteria. Rejected adding the action to projected Context nodes because issue #50 is about unscoped Workspace-level Contexts and reassignment is unsupported. Rejected hidden auto-assignment to a first/default Scope because Default Scope is superseded. Rejected a raw `window.prompt`/`window.confirm` because the codebase has a global dialog system and a native popup guard. Rejected direct React Flow node insertion after assignment because membership must come from Bus Scope Projection.

## Regression checklist
- Behavior: Workspace Home continues to show unscoped actor Contexts as `Workspace-level Contexts`, with copy avoiding “Default Scope” and “Default Field,” and already-scoped Contexts do not appear in that list or expose assignment.
- Behavior: Selecting/opening a Workspace-level Context still uses the existing actor conversation sidebar and still renders messages/activity correctly.
- Behavior: Assignment posts to `/v1/workspaces/:workspace_id/contexts/:context_id/assign-scope`, preserves the same `context_id`, removes the Context from Workspace-level Contexts after refresh, and makes it appear in the chosen Scope/Field projection via the existing projection API.
- Behavior: Missing Scope, already-scoped/invalid Context, and generic server/network failures are visible in structured UI states, preferably inline in the dialog/action area, and leave the dialog/action recoverable.
- Behavior: Scope Projection remains scoped-only; unscoped Contexts do not leak into any Field before assignment.
- Behavior: Field/canvas pan, zoom, selection, drag, node Open action, handles, connection affordances, rename, and Block Library drag/drop still pass existing regressions.

## Risk assessment
- Risk: Adding action controls inside the existing Workspace Context button can cause nested-button accessibility bugs or make the row click also open the conversation when assigning. Use sibling controls or stop propagation deliberately, and cover with Playwright.
- Risk: The current dialog component supports text input and checkbox but not select controls. For Scope selection, either implement the selection outside the confirmation dialog with normal React state, or extend the dialog deliberately with tests; do not fake selection through free-text Scope ids unless the UX clearly selects an existing Scope.
- Risk: Refresh ordering can leave stale UI if only Workspace-level Contexts refresh. Refresh the unscoped list, participant context list, and the target/open Scope Projection after success; handle the case where the selected/open context disappears from `workspaceContexts` but remains selected via participant contexts or projected opening.
- Risk: Existing Playwright fixtures and older specs still use `default`/`Default` as mocked Scope names in some places, while docs say Default Scope is not a product concept. Do not copy those fixtures into issue #50 tests; use real named Scopes. Consider future cleanup, but it is out of scope for this slice unless tests fail because of visible Default terminology.
- Mitigation: Keep implementation narrowly in Workspace Home and existing API/dialog/refresh paths; add tests for request identity, non-reassignment, failure mapping, no Default copy, and Field/canvas regressions.

## Decision confidence
- Confidence: high
- Reasons: The Bus assignment API and store behavior already exist and are well covered; Workspace Home already fetches unscoped Contexts separately; Scope Projection already derives scoped Context nodes from Bus; existing dialog and refresh patterns are clear. The main design choice is UI placement/selection mechanics, not substrate ownership.
- Open questions: Whether the product should automatically navigate to the target Field after assignment or simply refresh and leave the user on Workspace Home with a success state. Both can satisfy acceptance; recommended default is refresh in place and provide an “Open Field”/target Scope affordance after success to avoid surprising navigation.
